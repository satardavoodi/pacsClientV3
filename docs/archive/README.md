# Archive

Archived documents are kept for historical traceability, not as the primary source of truth.

## Subdirectories

| Folder | Contents |
|--------|----------|
| `performance-history/` | Historical performance analysis, bottleneck reports, Mode A/B architecture reviews |
| `design-proposals/` | Completed or superseded design proposals and task implementation guides |
| `bug-analysis/` | Historical bug analysis, code review reports, viewer tab audits |
| `ui-backups/` | UI snapshot documentation for specific versions |
| `module-framework/` | Superseded module execution planning and delivery docs |
| `root-guides/` | One-off implementation guides and older engineering notes |
| `../releases/archive/` | Version-by-version release notes and commit message artifacts |

## Reading Archived Notes Safely

- Expect stale paths, version numbers, and implementation assumptions.
- Prefer the documents in `docs/architecture/`, `docs/pipelines/`, `docs/stability/`, `docs/performance/`, `docs/modules/`, and `docs/development/` first.
- Treat archive content as background context, not current architecture.
