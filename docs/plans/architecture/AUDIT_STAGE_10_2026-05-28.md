## 1. Stage 10 strategy

`_hp_patient_open.py` installs a module-level rebind at lines 13–16:

```python
def print(*args, **_kw):
    _print_logger.debug(' '.join(str(a) for a in args))
```

The rebind serves a real purpose — it forces all `print()`-style debug output through the async queue handler instead of synchronously writing to console (slow on Windows). **The rebind itself is intentional and stays.**

The bug: every `print()` call lands at DEBUG level, which is below the `app.log` INFO threshold, so **error-path messages were invisible in the file**.

The fix: for **error paths**, route directly through `_logger.error(...)` / `_logger.warning(...)` — these calls **bypass the rebind** (because they don't use `print()` at all) and emit at INFO+ level, so they reach `app.log`. For **success-path workflow traces** (the `✅ [TAB] Activated tab at index N` lines), keep `print()` → DEBUG: they're high-frequency and would flood the file if promoted.

## 3. Safety

- **Zero behavior change on the success path.** The 4 success-trace `print()` calls still hit the rebind → DEBUG → console handler. Exactly what they did before.
- **On the error paths, messages now reach `app.log`.** Previously they were invisible in the file (only visible in the VS Code terminal via the console handler).
- **`exc_info=True`** on the 11 explicit-error calls means the stack trace lands in `app.log` too. Crucial for production debugging.
- **`%s` lazy formatting** used throughout (instead of f-strings) so the format work is skipped if the log level filters the record out — small but real performance improvement.
